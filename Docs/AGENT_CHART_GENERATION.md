# Agent Instructions: Generating SCC Chart JSON

This document tells an AI agent how to programmatically generate a valid chart JSON file that can be imported into the TC2 Single-Case Chart (SCC) application via the "Import Chart" button on the `/new` page.

---

## Quick Start

1. User describes the data, chart type, and desired annotations (lines, events, trends)
2. Agent generates the JSON using the schema below (or uses the helper script at `scripts/build/chart_generator.py`)
3. User imports the `.json` file via the app's Import Chart button

The import pathway (`importChart()` in `chartStorage.js`) will:
- Assign a fresh UUID and encryption key
- Set `shared: false`, assign the user's public key
- Run schema migrations
- Store to IndexedDB

You do NOT need to generate `id`, `chartKey`, `publicKey`, or `ownerName` â€” the app overwrites these on import.

---

## Native Format Detection

The app detects a "native TC2 format" JSON by checking for the presence of all three keys: `id`, `chartKey`, and `shared`. Your generated JSON MUST include these (use placeholder values â€” they get overwritten):

```json
{
  "id": "placeholder",
  "chartKey": "placeholder",
  "shared": false,
  ...
}
```

If these keys are missing, the import rejects the file as "Unrecognized JSON format."

## Complete Schema

### Root Object

```jsonc
{
  // === REQUIRED FOR FORMAT DETECTION (overwritten on import) ===
  "id": "placeholder",
  "chartKey": "placeholder",
  "shared": false,

  // === CHART IDENTITY ===
  "chartName": "Gold Prices 2020-2025",          // REQUIRED - display name
  "chartType": "Daily",                           // REQUIRED - see Chart Types below
  "minuteChart": false,                           // REQUIRED - true=frequency, false=raw counts
  "hasTimestamps": true,                          // REQUIRED - always true for agent-generated charts
  "_schemaVersion": 6,                            // REQUIRED - current schema version

  // === DATE ANCHOR ===
  "startDate": "2020-01-06T00:00:00.000Z",       // REQUIRED - ISO 8601 string, see Date System

  // === CHART DIMENSIONS ===
  "chartCapacity": 280,                           // Max x-positions (see Chart Types)
  "chartWindow": 140,                             // Visible window (<= capacity)

  // === DATA SERIES ===
  "series": { ... },                              // REQUIRED - see Data Series

  // === DRAWN LINES (all optional, default to {}) ===
  "PhaseLines": { ... },
  "AimLines": { ... },
  "LineCuts": { ... },
  "CelLines": { ... },

  // === VISIBILITY & STYLING (all optional, have sane defaults) ===
  "lineVisibility": { ... },
  "seriesVisibility": { ... },
  "traceStyles": { ... },
  "lineStyles": { ... },
  "legend": { ... },
  "fanVisible": true,
  "placeZerosBelowFloor": false,

  // === CREDITS (optional) ===
  "credits": {
    "0": "Source: Federal Reserve Economic Data (FRED)",
    "1": "Generated by AI agent"
  },

  // === METADATA (optional) ===
  "tags": ["gold", "commodities", "economics"],
  "acceptingEdits": false,
  "collaborators": [],
  "containerHeight": null
}
```

### Chart Types

| chartType              | capacity | yMin     | yMax      | unit | startDate alignment          |
|------------------------|----------|----------|-----------|------|------------------------------|
| `"Daily"`              | 280      | 0.69     | 1,000,000 | 7    | Previous Monday              |
| `"Weekly"`             | 200      | 0.00069  | 1,000     | 5    | Monday <= 1st of prev month  |
| `"Monthly"`            | 240      | 0.00069  | 1,000     | 6    | Jan 1 of previous year       |
| `"Yearly"`             | 200      | 0.00069  | 1,000     | 5    | Jan 1 of decade start        |
| `"FrequencyCollections"` | 280    | 0.00069  | 1,000     | 7    | Previous Monday              |

**The `unit` field** is the number of x-axis positions per doubling period. This matters for celeration line calculations.

**Y-axis is logarithmic.** All y-values must be positive numbers. The chart renders on a log scale between yMin and yMax. Values outside this range won't display.

### Minute Charts vs Count Charts

- **`minuteChart: true`** (frequency mode): Data entered as count + timing. Frequency = count / timing. The timing floor trace is displayed.
- **`minuteChart: false`** (count mode): Data entered as raw counts. Timing is internally set to 1.0 for all points. Frequency = count / 1 = count.

**For agent-generated data where you have pre-calculated values:**
- Use `minuteChart: false` (count mode)
- Put your values directly in `corrects` (or `errors`, `misc`)
- Set all `timing` values to `1` (or omit â€” compact format fills with `1`)
- The chart will display the raw values as-is

**Important:** Values are displayed on a **log scale**. A value of 0 cannot be plotted on a log scale. The app handles zeros by either placing them below the floor (at `threshold * 0.75`) or hiding them, controlled by `placeZerosBelowFloor`. **Always set `placeZerosBelowFloor` to `false`** in agent-generated charts â€” zeros should be hidden, not displayed as artificial below-floor points.

## Date System

### Timestamps

All dates in `series.xValues` are **Unix seconds** (not milliseconds):

```python
import datetime
timestamp = int(datetime.datetime(2024, 3, 15).timestamp())
# or: int(time.mktime(datetime.date(2024, 3, 15).timetuple()))
```

### startDate Alignment

`startDate` is the chart's anchor point â€” the date that maps to x-position 0. It MUST be aligned according to chart type:

| Chart Type | Alignment Rule | Example |
|------------|----------------|---------|
| Daily | Previous Monday (at or before earliest data) | Data starts 2024-03-15 (Fri) -> startDate = 2024-03-11 (Mon) |
| Weekly | Monday at or before 1st of previous month | Data starts 2024-03-15 -> prev month = Feb 1 -> Monday before = 2024-01-29 |
| Monthly | Jan 1 of previous year | Data starts 2024-03-15 -> startDate = 2023-01-01 |
| Yearly | Jan 1 of decade start | Data starts 2024 -> startDate = 2020-01-01 |

**`startDate` is stored as an ISO 8601 string** in the JSON: `"2024-03-11T05:00:00.000Z"` (midnight local time, serialized to UTC).

### Date-to-X-Position Conversion

The app converts timestamps to x-positions based on chart type:

| Chart Type | Formula |
|------------|---------|
| Daily | `x = daysDiff(date, startDate)` |
| Weekly | `x = floor(daysDiff(date, startDate) / 7)` |
| Monthly | `x = monthsDiff(date, startDate)` |
| Yearly | `x = yearsDiff(date, startDate)` |

**You do NOT need to pre-compute x-positions.** When `hasTimestamps: true`, the app converts timestamps to x-positions automatically during rendering. Just provide Unix-second timestamps in `series.xValues`.

### Date Boundary Policy (for line placement)

When specifying dates for lines, snap to boundaries:
- Daily: the day itself
- Weekly: Monday of that week
- Monthly: 1st of month
- Yearly: January 1st

## Data Series

```jsonc
"series": {
  "xValues": [1710460800, 1710547200, 1710633600, ...],  // Unix seconds
  "corrects": [42, 38, null, 45, ...],                    // Primary data (null = missing)
  "errors": [3, 5, null, 2, ...],                         // Secondary data (null = missing)
  "timing": [10, 10, null, 10, ...],                      // Minutes (or 1 for count charts)
  "misc": {
    "misc1": [100, 102, null, 98, ...],                   // Optional additional series
    "misc2": [50, 51, null, 49, ...]                      // Up to misc10
  }
}
```

### Rules

1. **All arrays must have equal length** â€” same length as `xValues`
2. **Use `null` for missing data** â€” not `0`, not `undefined`, not `NaN`
3. **Timestamps must be Unix seconds** â€” `Math.floor(date.getTime() / 1000)`
4. **Values must be non-negative numbers or null** â€” the log scale cannot display negatives
5. **For count charts**: set `timing` to all `1` values (or omit entirely â€” the compact format expansion fills missing `timing` with `null`, which the render pipeline handles)

### Compact Format

The app supports a compact format where:
- An entirely-null array can be omitted for `corrects`, `errors`, `timing` (e.g., if `errors` is all null, just don't include it)
- A constant array can be replaced with the scalar value (e.g., `"timing": 1` instead of `"timing": [1, 1, 1, ...]`)

The `expandChart()` function restores full arrays on load.

**CRITICAL: `series.misc` must ALWAYS be present**, even as an empty object `{}`. The app does `Object.keys(chartState.series.misc)` and `Object.entries(chartState.series.misc)` without null-checking. If `misc` is missing, the chart crashes on load.

### Mapping Real-World Data

For datasets like stock prices, temperature readings, etc.:

| Your Data | Map To | Rationale |
|-----------|--------|-----------|
| Primary values (price, count, etc.) | `corrects` | Main data series, rendered as filled circles |
| Secondary values | `errors` | Rendered as X markers |
| Additional series | `misc.misc1` - `misc.misc10` | Custom symbols/colors |
| Observation duration | `timing` | Minutes of observation (use `1` for pre-computed values) |

### Marker Symbols

Plotly marker symbols: `"circle"`, `"square"`, `"triangle-up"`, `"triangle-down"`, `"diamond"`, `"star"`, `"hexagon"`, `"pentagon"`, `"cross"`, `"hexagon2"`, `"octagon"`, `"x"`, `"none"` (invisible marker)

### Default Misc Colors & Symbols

When creating misc series, cycle through these defaults:

```
Colors:  #FFA500, #FF0000, #00AA00, #0000FF, #FF00FF, #00CCCC, #800080, #008080, #FFD700, #8B4513
Symbols: square, triangle-up, diamond, star, hexagon, pentagon, cross, triangle-down, hexagon2, octagon
```

### Aggregation Options (`onXAgg`)

When multiple data points share the same x-position: `"raw"`, `"median"`, `"mean"`, `"min"`, `"max"`, `"first"`, `"last"`, `"sum"`

### Rolling Window (`acrossXAgg`)

```jsonc
"acrossXAgg": { "fn": "median", "window": 7 }   // 7-position rolling median
// or
"acrossXAgg": null                                 // No rolling window
```

## Phase Lines

Phase lines mark condition changes. Each has a vertical line, a horizontal line, and a text label.

```jsonc
"PhaseLines": {
  "phase-1": {
    "id": "phase-1",
    "direction": "top",                              // "top" = line drops from top, "bottom" = rises from bottom
    "verticalLineDate": "2024-06-15T00:00:00.000Z",  // ISO string: where vertical line is
    "verticalLineY": 100,                             // Y value where horizontal meets vertical
    "horizontalEndDate": "2024-09-15T00:00:00.000Z",  // ISO string: where horizontal line ends
    "text": "Fed Rate Cut",                           // Label text
    "style": {
      "color": "black",
      "width": 2,
      "dash": "solid",
      "fontColor": "black",
      "fontSize": 14
    },
    "shapeIndices": [],                               // Leave empty - populated at render time
    "annotationIndex": -1                             // Leave -1 - populated at render time
  }
}
```

**Usage for events/phase changes:** Place a phase line at each significant event. The vertical line marks the date, the horizontal line spans the phase duration, and the text labels what changed.

## Aim Lines

Aim lines are target/goal lines. Can be horizontal (constant target) or diagonal (trend target).

```jsonc
"AimLines": {
  "aim-1": {
    "id": "aim-1",
    "direction": "horizontal",                       // "horizontal" or "diagonal"
    "date1": "2024-01-01T00:00:00.000Z",             // Start date (ISO string)
    "y1": 2000,                                       // Start y-value
    "date2": "2024-12-31T00:00:00.000Z",             // End date (ISO string)
    "y2": 2000,                                       // End y-value (same as y1 for horizontal)
    "text": "Target: $2000",                          // Label text (optional, "" for no label)
    "style": {
      "color": "black",
      "width": 2,
      "dash": "dash",
      "fontColor": "black",
      "fontSize": 14
    },
    "shapeIndices": [],
    "annotationIndex": -1
  }
}
```

**For diagonal aim lines:** Set different y1/y2 values. The line will slope between them.

## Cut Lines

Cut lines segment data for separate trend analysis. They create vertical boundaries.

```jsonc
"LineCuts": {
  "cut-1": {
    "id": "cut-1",
    "date": "2024-06-01T00:00:00.000Z"               // ISO string
  }
}
```

**Important:** Cut line dates are used as Date objects internally via `cut.date.getTime()`. After JSON import and IDB roundtrip, these remain as ISO strings. The app's `loadChart` -> `deserializeChart` does not explicitly convert line dates back to Date objects. This may cause issues for charts with cut lines after import. Phase lines and aim lines use dates only for shape positioning at draw time and may handle string dates differently.

**Recommendation:** For maximum reliability, prefer using phase lines (which also mark boundaries visually) over cut lines in agent-generated charts. If you need cut lines, test that import works correctly with the current app version.

## Celeration Lines (Trend Lines)

Celeration lines show rate of change on the log scale. They are fitted to data between two dates.

```jsonc
"CelLines": {
  "cel-1": {
    "id": "cel-1",
    "seriesKey": "corrects",                          // Which series this trend tracks
    "aggId": "0",                                      // Which aggregation of that series
    "date1": "2024-01-01T00:00:00.000Z",              // Trend start (ISO string)
    "y1": 50,                                          // Y at start (display scale, not log)
    "date2": "2024-06-30T00:00:00.000Z",              // Trend end (ISO string)
    "y2": 200,                                         // Y at end
    "bounceUpperY1": null,                             // Upper bounce envelope (null = none)
    "bounceUpperY2": null,
    "bounceLowerY1": null,                             // Lower bounce envelope (null = none)
    "bounceLowerY2": null,
    "text": "x2.0",                                    // Celeration value label
    "fitMethod": "",                                   // Deprecated (stored in user prefs)
    "bounceEnvelope": "",                              // Deprecated
    "style": {
      "color": "green",                                // green=corrects, red=errors, orange=timing
      "width": 2,
      "dash": "solid",
      "bounceColor": "green",
      "bounceWidth": 1,
      "bounceDash": "dash"
    },
    "shapeIndices": [],
    "annotationIndex": -1
  }
}
```

### Calculating Celeration Values

Celeration = the multiply factor per `unit` x-positions (see Chart Types table).

On a Daily chart (`unit=7`, so per week):
- If the value doubles every 7 days: celeration = x2.0
- The line goes from (x1, y1) to (x2, y2) on the log scale

To compute y2 from a desired celeration:
```
y2 = y1 * (celeration ^ (xSpan / unit))
```

Where `xSpan` = number of x-positions between date1 and date2.

## Step-by-Step Agent Workflow

### 1. Gather Requirements

From the user prompt, determine:
- **Data source**: What data to chart (prices, counts, measurements)?
- **Date range**: What time period?
- **Chart type**: Daily (default), Weekly, Monthly, or Yearly?
- **Minute vs count**: Is there a timing/duration component, or raw values?
- **Series mapping**: Which data goes in corrects, errors, misc?
- **Events/phases**: Any significant events to mark with phase lines?
- **Targets**: Any target levels to mark with aim lines?
- **Trends**: Any trend lines (celeration lines) to fit?

### 2. Obtain the Data

- Use web search, APIs, or user-provided data
- Structure as parallel arrays: dates[], values[], ...
- Handle missing data as `null`

### 3. Generate the JSON

Use the helper script (`scripts/build/chart_generator.py`) or build manually:

```python
from scripts.build.chart_generator import ChartBuilder

chart = ChartBuilder("Gold Prices 2020-2025", chart_type="Daily", minute_chart=False)

# Add data
for date, price in gold_data:
    chart.add_point(date, corrects=price)

# Add events as phase lines
chart.add_phase_line("2020-03-11", "2020-06-01", "COVID Crash", direction="top", y=1800)
chart.add_phase_line("2022-03-16", "2022-12-01", "Rate Hikes Begin", direction="top", y=2000)

# Add target
chart.add_aim_line("2020-01-01", "2025-12-31", y=2000, text="$2000 Target")

# Export
chart.save("gold_prices.json")
```

### 4. Validate

Before delivering the JSON, verify:
- [ ] `id`, `chartKey`, `shared` present (for format detection)
- [ ] `_schemaVersion` is 6
- [ ] `startDate` is properly aligned for chart type
- [ ] All series arrays have the same length as `xValues`
- [ ] No negative values in data (log scale)
- [ ] `chartCapacity` matches chart type
- [ ] `traceStyles` has entries for all used series
- [ ] Line dates are ISO 8601 strings
- [ ] `hasTimestamps` is `true`

### 5. Deliver

Save as `.json` file. The user imports via the app's "Import Chart" button on `/new`.

## Y-Axis Scale Reference

The chart uses a **logarithmic y-axis**. Data values map directly to positions on this scale.

| Chart Type | Y Range | Example Values |
|------------|---------|----------------|
| Daily | 0.69 - 1,000,000 | Counts from <1 to 1M per minute |
| Weekly/Monthly/Yearly | 0.00069 - 1,000 | Lower frequencies over longer periods |

For non-behavioral data (prices, temperatures, etc.), the values just need to be positive and within the y-range. If values exceed the range, they'll be clipped.
