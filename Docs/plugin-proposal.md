# TC2 Chart Interface as a Portable Plugin

## Goal

Extract TC2's Standard Celeration Chart interface into a self-contained JavaScript plugin that any web app can import and use. The host app passes in data, gets a fully interactive SCC chart — all line tools, annotations, celeration lines, data entry, series controls — without needing to know anything about TC2's internals.

Immediate consumer: **FL-Flash2**, a flashcard app that wants to display learning session data on SCC charts. Future consumers: any app in this ecosystem, or external apps entirely.

The end result looks like this from the consumer's perspective:

```javascript
import { SCCChart } from 'scc-chart.js';

const chart = new SCCChart(containerElement, {
    chartType: 'Daily',
    minuteChart: true,
    series: {
        xValues: [...],
        corrects: [...],
        errors: [...],
        timing: [...]
    }
});

chart.on('dataChanged', (state) => {
    // host app persists however it wants
});
```

---

## Why

TC2 is already the right tool for SCC charting. The rendering engine, interaction tools, and celeration math are all there. The problem is that the chart interface is entangled with TC2's own storage, sync, and page structure — making it impossible to drop into another app without carrying all of TC2 along with it.

Extracting it as a plugin:
- Lets FL-Flash2 (and future apps) use TC2's chart interface without iframes or code duplication
- Makes TC2's own app a consumer of the plugin, keeping one source of truth
- Opens the chart interface to use cases beyond TC2's own sync/storage model

---

## What the Plugin Owns

- All chart rendering (Plotly.js, SCC layout, axis logic)
- All interaction tools (phase lines, aim lines, cut lines, celeration lines)
- All series controls (data entry, misc series, visibility toggles)
- Its own HTML structure and styles for the chart UI

## What the Plugin Does Not Own

- Storage (host app decides where data lives)
- Authentication / encryption
- Sync

---

## First Step: Refactor TC2 to Support This

Before packaging, TC2's chart code needs to be refactored to support clean extraction. Two things currently prevent it:

1. **Scattered initialization** — starting a chart requires calling multiple functions across multiple files in sequence. This needs to collapse into a single entry point.

2. **Hardcoded DOM assumptions** — UI controls are wired up by hardcoded element IDs that assume TC2's own page structure. These need to either be generated by the plugin itself or be configurable.

The event system and storage decoupling are already well-designed and need minimal changes.

---

## Questions for Assessment

The following questions need to be answered by reading TC2's source before a detailed refactor plan can be made.

### DOM & HTML Structure

1. Which hardcoded element IDs belong to controls that are visually part of the chart interface (and therefore the plugin should render itself), vs. controls that belong to TC2's app shell (nav, sync UI, account management, etc.)?
2. Is there a single HTML template file that defines TC2's chart page? If so, what is the full DOM structure — which elements are chart UI vs. app shell?
3. Are any element IDs referenced in more than one JS file? List them — these are the highest-risk ones to refactor.
4. Does any JS file reference IDs that belong to elements outside the chart page entirely (e.g., global nav, welcome screen)?
5. How much of TC2's CSS is scoped to the chart interface vs. the overall app? Is there a separation already?

### Initialization & Dependencies

6. What is the exact sequence of initialization calls required to render a chart, and what are the dependencies between them? Which calls can be skipped when there is no storage or sync?
7. Are there any circular imports between modules?
8. Is `chartState` a singleton? If two charts were on the same page, would they conflict?
9. Is the event bus a singleton? Same question — would two chart instances on the same page produce event collisions?
10. Which modules import from the storage or sync layer directly (not via events)? List all direct imports of `chartStorage.js`, `syncClient.js`, and `crypto.js` from non-storage files.

### Public API Surface

11. What is the minimum `chartState` shape needed to render a blank chart with no data?
12. What events would a host app need to subscribe to in order to know when any user-driven state change has occurred (data entry, line drawn, line deleted, settings changed)? Is the current event set sufficient, or are there gaps?
13. When a user modifies data (adds a data point, draws a line), what does the full updated state look like that the host app would need to persist? Is there a single serializable object, or does the host need to reconstruct it from multiple sources?

### Packaging

14. Does TC2 currently use any build tooling (webpack, esbuild, rollup)? Or are modules loaded directly as ES6 imports?
15. Are there any non-JS assets (CSS files, fonts, SVGs) that the chart interface depends on? Where are they and how are they referenced?
16. Does the chart interface use any browser APIs beyond IndexedDB (Web Crypto, Service Worker, etc.) that would need to be stripped or made optional?
